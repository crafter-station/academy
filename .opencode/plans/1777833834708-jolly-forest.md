# Crafter Academy Landing Page + Polar Checkout Plan

## Goal

Replace the starter page with a polished editorial-style Crafter Academy landing page and add a Polar checkout for the USD 999 cohort seat.

Confirmed decisions:
- Polar checkout uses `POLAR_ACCESS_TOKEN` and `POLAR_PRODUCT_ID` from environment variables.
- Scholarship CTA points to `academy@crafterstation.com`.
- The page is the root route, `/`.

Note: no Polar skill is installed in this environment. The plan uses Polar's official docs for the TypeScript SDK and Next.js route handlers.

## Implementation

### 1. Dependencies

Modify `package.json` and `bun.lock` by installing Polar's TypeScript SDK:

```bash
bun add @polar-sh/sdk
```

Use the direct SDK instead of a client component so the landing page CTA can be a normal link to a server route.

### 2. Environment Example

Add `.env.example` with:

```env
POLAR_ACCESS_TOKEN=
POLAR_PRODUCT_ID=
POLAR_SERVER=production
NEXT_PUBLIC_SITE_URL=http://localhost:3000
```

`POLAR_PRODUCT_ID` should be a one-time Polar product configured at USD 999. `POLAR_SERVER` can be `sandbox` for testing; otherwise production is used.

### 3. Polar Checkout Route

Add `src/app/checkout/route.ts`.

Recommended shape:
- `export const runtime = "nodejs"`.
- Handle `GET` so CTA links work without client-side JavaScript.
- Validate `POLAR_ACCESS_TOKEN` and `POLAR_PRODUCT_ID`; return a clear `500` response if missing.
- Create a checkout session with `new Polar({ accessToken, server })` and `polar.checkouts.create(...)`.
- Pass `products: [POLAR_PRODUCT_ID]`.
- Use a success URL shaped like `https://site.example/?checkout=success&checkout_id={CHECKOUT_ID}`.
- Use `returnUrl: NEXT_PUBLIC_SITE_URL`.
- Add metadata such as `{ product: "crafter-academy", cohort: "1" }`.
- Redirect to the returned `checkout.url`.

This keeps the product ID server-side, avoids exposing the access token, and avoids adding a client checkout button.

### 4. Landing Page

Replace `src/app/page.tsx` entirely.

Use local arrays and small local components in the same file unless the file becomes hard to scan. Avoid adding reusable abstractions until needed.

Content sections:
- Masthead with `Crafter Academy`, cohort proof, and a `Reserve your seat` link to `/checkout`.
- Hero built around: `Building is cool. Shipping is everything.`
- Editorial lede explaining Crafter Station, 600+ LATAM builders, 5 years of learning, and the shift from engineering-only to distribution-led product building.
- Proof strip with companies and communities: Rappi, Nubank, Scotiabank, Kebo, Midudev, 30x, Clerk, Zabio, and more.
- Program snapshot: 20 crafters, 10 weeks, 2 live classes/week, 5 mentors, 4 mentees per mentor, 2 challenges/week, demo days in weeks 5 and 10, USD 999.
- Who it is for: people with programming knowledge who can understand a recursive Fibonacci function and want to ship real products.
- Curriculum arc: Product Thinking, Building, Shipping.
- Operating system: choose a project in week 1, private mentor-reviewed challenge, public build-in-public challenge, WhatsApp mentor groups, weekly 1-1 mentor availability, demo day checkpoints.
- Syllabus preview: compress the 10-week plan into an editorial timeline and link to the provided Notion syllabus.
- Mentorship section: emphasize 5 mentors, small groups, feedback, and connections to builders/operators.
- Scholarships section: `10 scholarships for promising builders`, with `mailto:academy@crafterstation.com?subject=Crafter%20Academy%20Scholarship`.
- Pricing section: USD 999, explain the class economics, and show what is included.
- FAQ section with native `<details>` blocks to avoid client JS.
- Final CTA: `20 crafters. 10 weeks. Ship something real.` with `/checkout` and scholarship email links.

FAQ questions to include:
- Who is Crafter Academy for?
- Do I need to know how to code?
- What will I build?
- What is taught?
- How do the private and public challenges work?
- How does mentorship work?
- What happens on demo days?
- How much does it cost?
- Are scholarships available?
- How do I apply for a scholarship?
- Is this only for LATAM?
- What if I miss a live class?

Visual direction:
- Warm off-white editorial background, graphite text, one saturated ochre/oxblood accent.
- Large serif-like editorial headings and quiet sans/mono metadata.
- Asymmetric grids, hairline borders, section numbers, pull quote, dense but readable cards.
- No generic SaaS gradients or dashboard mockups.
- Keep desktop and mobile layouts first-class.

### 5. Metadata and Typography

Modify `src/app/layout.tsx`:
- Update metadata to Crafter Academy title and description.
- Add a high-contrast Google serif such as `Libre_Baskerville` via `next/font/google` if build-compatible.
- Add its variable to `<html>`.

Modify `src/app/globals.css` minimally:
- Map `--font-heading` to the new serif variable if added.
- Keep the existing Tailwind v4/shadcn token setup intact.

If the added Google serif causes build or font-loading issues, keep the existing fonts and use system serif utility classes in the page instead.

## Verification

Run:

```bash
bun run lint
bunx tsc --noEmit
bun run build
```

Manual checks:
- Start with `bun run dev`.
- Visit `/` on desktop and mobile widths.
- Confirm all sections render and the page feels editorial, not generic SaaS.
- Confirm the scholarship CTA opens an email to `academy@crafterstation.com`.
- With missing Polar env vars, confirm `/checkout` returns a clear configuration error.
- With valid Polar env vars, confirm `/checkout` redirects to Polar checkout for the USD 999 product.
- Confirm Polar success returns to `/?checkout=success&checkout_id=...`.

## Risks

- The USD 999 price must be configured in Polar for `POLAR_PRODUCT_ID`; the app will not create or mutate products.
- No webhook or database fulfillment is planned because the user only asked for checkout. Seat-cap enforcement remains operational or Polar-side.
- Exact Polar SDK response types should be verified by TypeScript after installing `@polar-sh/sdk`.
