# Plan: Switch Enrollment Checkout From Polar To Stripe

## Goal

Replace Polar with Stripe while preserving the existing paid-user onboarding flow:

- `/checkout` remains the public checkout entrypoint used by existing CTAs.
- Stripe Checkout handles a one-time payment using a manually created dashboard Price ID.
- Stripe webhook verification is the source of truth for fulfillment.
- Verified paid checkout creates or updates a Clerk user by checkout email.
- Clerk `privateMetadata.enrollment.status = "paid"` remains the access gate for `/enrolled`.
- Resend sends the same welcome email inviting the buyer to open `https://academy.crafter.run/enrolled` and sign in with the checkout email.
- Browser success redirects never provision access.

## Recommended Flow

1. User clicks an existing CTA pointing to `/checkout`.
2. `src/app/checkout/route.ts` creates a Stripe Checkout Session with `mode: "payment"` and `STRIPE_PRICE_ID`.
3. Stripe redirects the browser to `/checkout/success?session_id={CHECKOUT_SESSION_ID}` after payment.
4. `/checkout/success` stays a neutral confirmation page. It can link to `/enrolled`, but it does not grant access.
5. Stripe sends `checkout.session.completed` to `/api/stripe/webhook`.
6. The webhook verifies `stripe-signature`, validates the session is paid and for the configured price, then provisions the Clerk user and sends the welcome email.
7. User opens `/enrolled`; Clerk redirects signed-out users to `/sign-in`.
8. User signs in with email code using the checkout email and sees the enrolled page.

## Security And Idempotency

- Never create Clerk users from `/checkout/success` or any browser query param.
- Verify Stripe webhook signatures before reading event data.
- Validate session mode, paid status, product/cohort metadata, and actual line item price ID.
- Use Clerk user lookup by email before creating a user.
- Use Clerk metadata as lightweight idempotency storage because there is no DB.
- If the same Stripe session is retried and metadata already has `welcomeEmailSentAt`, return success without resending email.
- If access metadata is written but Resend fails, let the webhook fail so Stripe retries and can send the welcome email later. Access remaining paid is acceptable.
- Exactly-once email delivery is best-effort without a database or queue.
